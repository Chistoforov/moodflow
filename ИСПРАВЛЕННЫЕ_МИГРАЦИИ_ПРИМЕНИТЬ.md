# ⚡ ИСПРАВЛЕННЫЕ МИГРАЦИИ (используйте это!)

## ❗ Проблема с предыдущей версией
В таблице `psychologists` поле `user_id` имеет тип **TEXT**, а не UUID.
Это вызывало ошибку: `operator does not exist: text = uuid`

## ✅ Исправление
Нужно делать JOIN через таблицу `users` для правильного сравнения типов.

### Шаг 2: Выполните ИСПРАВЛЕННЫЙ SQL (Миграция 024)

**Скопируйте и выполните:**

```sql
-- ========================================
-- МИГРАЦИЯ 024: Создание таблицы (ИСПРАВЛЕННАЯ)
-- ========================================

CREATE TABLE IF NOT EXISTS public.monthly_analytics (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  year INTEGER NOT NULL,
  month INTEGER NOT NULL,
  week_number INTEGER NOT NULL,
  days_analyzed INTEGER NOT NULL,
  analysis_text TEXT NOT NULL,
  general_impression TEXT,
  positive_trends TEXT,
  decline_reasons TEXT,
  recommendations TEXT,
  reflection_directions TEXT,
  perplexity_request_id TEXT,
  perplexity_response JSONB,
  is_final BOOLEAN DEFAULT false,
  status TEXT DEFAULT 'completed' CHECK (status IN ('pending', 'processing', 'completed', 'failed')),
  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  UNIQUE(user_id, year, month, week_number)
);

CREATE INDEX IF NOT EXISTS idx_monthly_analytics_user_date ON public.monthly_analytics(user_id, year, month);
CREATE INDEX IF NOT EXISTS idx_monthly_analytics_final ON public.monthly_analytics(user_id, year, month, is_final) WHERE is_final = true;

ALTER TABLE public.monthly_analytics ENABLE ROW LEVEL SECURITY;

-- Policy: Users can read their own analytics
CREATE POLICY "Users can read their own monthly analytics"
  ON public.monthly_analytics
  FOR SELECT
  USING (auth.uid() = user_id);

-- Policy: Service role can manage
CREATE POLICY "Service role can manage monthly analytics"
  ON public.monthly_analytics
  FOR ALL
  USING (auth.jwt()->>'role' = 'service_role');

-- Policy: Users can insert their own
CREATE POLICY "Users can insert their own monthly analytics"
  ON public.monthly_analytics
  FOR INSERT
  WITH CHECK (auth.uid() = user_id);

-- ИСПРАВЛЕННАЯ политика: JOIN через users для правильного сравнения типов
CREATE POLICY "Admins can read all monthly analytics"
  ON public.monthly_analytics
  FOR SELECT
  USING (
    EXISTS (
      SELECT 1 FROM public.psychologists p
      JOIN public.users u ON p.user_id = u.sso_uid
      WHERE u.id = auth.uid()
      AND p.role = 'admin'
      AND p.active = true
    )
  );

-- Function для updated_at
CREATE OR REPLACE FUNCTION update_monthly_analytics_updated_at()
RETURNS TRIGGER AS $$
BEGIN
  NEW.updated_at = NOW();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

-- Trigger
CREATE TRIGGER update_monthly_analytics_updated_at
  BEFORE UPDATE ON public.monthly_analytics
  FOR EACH ROW
  EXECUTE FUNCTION update_monthly_analytics_updated_at();
```

**Нажмите RUN** ✅

## 📝 Что было исправлено

### Было (❌ ошибка):
```sql
WHERE user_id = auth.uid()
-- user_id (TEXT) = auth.uid() (UUID) ❌
```

### Стало (✅ работает):
```sql
WHERE u.id = auth.uid()
-- JOIN через users, сравниваем UUID с UUID ✅
```

## 🔍 Техническая справка

**Структура данных:**
- `auth.users.id` → UUID (из Supabase Auth)
- `users.id` → UUID (наша таблица)
- `users.sso_uid` → TEXT (связь с auth.users.id в виде строки)
- `psychologists.user_id` → TEXT (ссылается на users.sso_uid)

**Почему JOIN:**
```
auth.uid() (UUID) 
  → users.id (UUID) ✅
  → users.sso_uid (TEXT) 
  → psychologists.user_id (TEXT) ✅
```

Без JOIN нельзя напрямую сравнить UUID с TEXT!
